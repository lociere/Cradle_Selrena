## 一、架构核心铁律（绝对不可突破，所有模块必须100%遵守）

本架构所有设计严格遵循以下规则，从根源避免架构腐化、依赖污染、人设漂移、安全风险：

1. **灵魂与身体100%物理隔离**：Python AI层为纯灵魂核心，仅负责人格、记忆、情绪、思考、内容生成，**绝对不碰任何平台/场景/IO/硬件/网络逻辑**，唯一通信对象是TS内核，仅通过标准化IPC协议交互。

2. **核心完全封闭，扩展完全开放**：内核核心模块完全封闭，运行时不可篡改；所有扩展能力（平台接入、功能增强、技能扩展）仅能通过插件系统实现，插件仅可依赖全局协议，**绝对不能侵入内核内部、访问内核私有实例**。

3. **多语言完全隔离，无代码级依赖**：TypeScript、Python、C++三大技术栈的环境、依赖、生命周期完全独立，仅通过标准化协议通信，无任何代码导入、函数调用级依赖，互不污染、互不影响。

4. **人设终身稳定，记忆与知识完全分离**：核心人设配置注入后完全冻结，运行时不可修改；对话记忆、人设知识库、通用知识库三者物理隔离，**绝对不会出现记忆污染人设的问题**。

5. **最小权限原则，沙箱安全隔离**：每个模块/插件仅拥有完成自身职责所需的最小权限，插件运行在独立沙箱中，无授权绝对不能访问文件系统、网络、内核内部数据。

6. **全链路可观测，问题可追溯**：所有流程带唯一trace_id全链路透传，跨进程、跨模块、跨语言不中断，日志、事件、异常全链路绑定，100%可定位、可回溯。

7. **本地优先，隐私100%可控**：所有核心逻辑、模型推理、记忆存储100%运行在用户本地设备，无强制云端依赖，用户数据完全可控，无隐私泄露风险。

8. **幂等性设计，优雅生命周期**：所有模块的启动/停止必须实现幂等性，重复调用无副作用；停机时严格按依赖顺序优雅释放资源，保证数据不丢失、不损坏。

### 3.2 核心内核层（全模块补全，大脑中枢）

**核心定位**：整个项目的唯一大脑中枢，全项目生命周期总控、资源调度、IPC通信、插件管理、持久化、配置管理、安全管控，是连接所有模块的核心枢纽。核心模块完全封闭，仅通过全局协议暴露能力，运行时不可篡改。

#### 核心设计规范

1. **唯一可信源**：所有配置、记忆、状态的唯一可信源都在内核，其他模块仅能通过内核暴露的API获取/同步数据，绝对不能自行修改。

2. **生命周期总控**：全项目所有模块的启动/停止/重启，都由内核统一调度，严格按依赖顺序执行，保证启动/停机的幂等性、优雅性。

3. **完全隔离解耦**：内核内部模块通过依赖注入解耦，模块间仅通过事件总线通信，无硬编码依赖；外部模块绝对不能直接访问内核内部私有实例。

4. **配置注入唯一入口**：所有模块的配置都由内核统一加载、校验、注入，其他模块绝对不允许直接读取本地配置文件。

5. **安全管控总闸**：所有插件调用、跨进程通信、外部输入，都必须经过内核的权限校验、参数校验、安全审计，无授权的操作直接拦截。

#### 完整模块设计（全量补全）

##### 3.2.1 生命周期管理器

**核心职责**：统一调度全项目所有模块的启动/停止/重启，严格按依赖顺序执行，保证全流程的幂等性、优雅性，是内核启动的入口。

- **核心启动顺序（严格不可颠倒）**：

    1. 初始化全局日志与可观测性系统

    2. 加载全局配置，执行合法性校验，冻结核心人设配置

    3. 初始化本地持久化数据库，执行数据迁移与备份

    4. 初始化全局事件总线

    5. 启动C++原生加速模块，加载模型，校验算力可用性

    6. 启动IPC通信服务端，监听端口，等待客户端连接

    7. 启动Python AI层子进程，注入冻结配置，建立IPC连接，校验可用性

    8. 初始化插件管理器，执行插件清单校验、沙箱初始化，加载合规插件

    9. 启动生命时钟管理器，开始发送心跳

    10. 启动渲染层窗口，建立与渲染层的通信

    11. 发布「应用启动完成」全局事件，进入正常运行状态

- **核心停机顺序（严格不可颠倒）**：

    1. 发布「应用即将停机」全局事件，停止接收新的用户请求

    2. 停止生命时钟管理器，终止主动思维心跳

    3. 停止所有插件，执行插件卸载钩子，释放插件资源

    4. 发送停机指令给Python AI层，等待AI层优雅停机，全量同步记忆

    5. 执行全量数据持久化，提交数据库事务，执行备份

    6. 关闭IPC通信服务端，断开所有客户端连接

    7. 停止C++原生加速模块，释放模型显存/内存

    8. 关闭持久化数据库连接

    9. 关闭渲染层窗口，释放Electron资源

    10. 发布「应用已停机」事件，退出进程

- **核心接口**：

    ```TypeScript
    
    interface ILifecycleManager {
      start(): Promise<void>;
      stop(): Promise<void>;
      restartModule(moduleName: string): Promise<void>;
      getCurrentState(): AppLifecycleState;
    }
    ```

- **边界规则**：仅生命周期管理器可调度模块的启动/停止，其他模块绝对不能自行启动/停止其他核心模块。

##### 3.2.2 配置管理器

**核心职责**：全项目所有配置的唯一可信源，统一加载、校验、管理、分发配置，是所有模块配置的唯一入口。

- **核心功能**：

    1. 统一加载`configs/`目录下的所有配置文件，执行格式校验、类型校验、合法性校验，校验不通过直接终止启动。

    2. 核心人设配置、推理配置注入到Python AI层后，自动冻结，运行时不可修改，防止人设漂移。

    3. 非核心配置支持热重载，修改配置文件后自动更新，无需重启应用。

    4. 为每个模块提供仅读的配置快照，模块不能修改全局配置，仅能通过配置管理器申请修改。

    5. 配置修改自动记录审计日志，支持配置版本回溯。

|    AI推理配置|    `configs/ai/inference.yaml`|    ❌ 运行时冻结|    Python AI层、C++加速层|
|    内核配置|    `configs/kernel/`|    ✅|    内核各模块|
|    插件配置|    `configs/plugin/`|    ✅|    插件管理器、各插件|
|    全局通用配置|    `configs/general.yaml`|    ✅|    全模块|
- **边界规则**：所有模块绝对不能直接读取本地配置文件，所有配置必须从配置管理器获取，核心配置运行时完全冻结，不可修改。

##### 3.2.3 IPC通信管理器

**核心职责**：管理跨进程通信服务端，处理TS内核与Python AI层、C++加速层、渲染层的IPC通信，保证消息的有序性、可靠性、全链路追踪。

- **技术选型**：ZMQ零消息队列，REQ/REP同步请求响应模式 + PUB/SUB异步事件发布订阅模式，低延迟、高可靠，适配本地跨进程通信。

- **核心功能**：

    1. 启动IPC服务端，绑定固定地址，监听客户端连接，自动管理连接生命周期。

    2. 消息序列化/反序列化，核心消息用Protobuf序列化，保证跨语言兼容性、高性能。

    3. 全链路trace_id透传，所有消息必须带trace_id，跨进程不中断。

    4. 消息超时控制、自动重试、异常重连机制，保证通信可靠性。

    5. 消息格式校验、权限校验，不符合协议的消息直接拦截，记录审计日志。

    6. 流量控制、消息积压处理，防止内存溢出，保证内核稳定性。

- **核心消息类型全量定义**：

|    消息类型|    模式|    发送方|    接收方|    核心作用|
|---|---|---|---|---|
|    `config_init`|    REQ/REP|    TS内核|    Python AI层|    注入全局冻结配置，初始化AI层|
|    `chat_message`|    REQ/REP|    TS内核|    Python AI层|    传入标准化用户消息，获取AI回复|
|    `life_heartbeat`|    REQ/REP|    TS内核|    Python AI层|    生命时钟心跳，触发主动思维|
|    `memory_sync`|    PUB|    Python AI层|    TS内核|    同步长期记忆，触发持久化|
|    `state_sync`|    PUB|    Python AI层|    TS内核|    同步AI状态（情绪、唤醒状态等）|
|    `native_inference`|    REQ/REP|    TS内核|    C++加速层|    调用原生算力推理能力|
|    `render_state_sync`|    PUB|    TS内核|    渲染层|    同步状态给渲染层，驱动Live2D动作|
|    `user_interaction`|    REQ/REP|    渲染层|    TS内核|    接收用户界面交互事件|
- **边界规则**：所有跨进程通信必须经过IPC通信管理器，模块间不能直接建立跨进程连接；Python AI层仅能与TS内核通信，不能直接连接其他模块。

##### 3.2.4 持久化管理器

**核心职责**：全项目所有数据的唯一持久化入口，管理本地嵌入式数据库，其他模块绝对不能直接读写本地文件/数据库。

- **技术选型**：SQLite本地嵌入式数据库，无服务端依赖，单文件存储，适配桌面端场景，支持事务、加密、备份。

- **核心功能**：

    1. 数据库连接管理、事务控制、数据迁移、自动备份，保证数据一致性、安全性。

    2. 提供标准化的CRUD接口，所有模块的持久化操作必须通过此接口执行，禁止直接执行SQL。

    3. 敏感数据加密存储，保护用户隐私，防止数据泄露。

    4. 记忆的自动沉淀：会话结束后，自动筛选短期记忆中的重要内容，沉淀到长期记忆库。

    5. 记忆的自动遗忘：长期记忆权重自动衰减，低于阈值的记忆自动归档，符合人脑遗忘曲线。

    6. 操作审计日志全量记录，所有数据修改操作都可追溯。

- **核心数据库表设计**：

|    表名|    核心存储内容|    访问权限|
|---|---|---|
|    `long_term_memory`|    长期记忆库，所有记忆片段、权重、类型、标签|    仅内核可写，Python AI层仅可同步|
|    `preference_memory`|    用户偏好记忆，终身保留，优先级最高|    仅内核可读写|
|    `knowledge_base`|    通用知识库，与人设知识库分离|    仅内核可读写|
|    `config_snapshots`|    配置快照，配置修改历史，支持回溯|    仅内核可读写|
|    `plugin_configs`|    插件配置，每个插件的配置完全隔离|    插件仅可读写自身配置|
|    `app_state`|    应用状态，窗口位置、主题、用户偏好设置|    仅内核可读写|
|    `operation_audit_log`|    操作审计日志，全量记录所有数据修改|    仅内核可追加，不可修改|
- **边界规则**：Python AI层仅能通过`memory_sync`事件通知内核同步记忆，不能直接读写数据库；每个插件仅能读写自身的配置，不能访问其他插件/内核的配置；所有写操作必须在事务中执行，保证数据一致性。

##### 3.2.5 插件管理器（全量补全，用户重点关注）

**核心职责**：插件的全生命周期管理、沙箱隔离、权限管控、安全校验，是所有扩展能力的唯一入口，绝对保证插件不能侵入内核内部。

- **核心设计原则**：插件沙箱隔离、最小权限、崩溃隔离、热插拔、无侵入核心。

- **核心功能全量设计**：

    1. **插件清单校验**：

        - 所有插件必须提供`plugin-manifest.yaml`清单文件，定义插件名称、版本、作者、权限声明、依赖、入口文件。

        - 插件加载前，自动校验清单格式、合法性、权限声明，不符合规范的插件直接拦截，禁止加载。

        - 插件清单示例：

            ```YAML
            
            id: napcat-qq-adapter
            name: QQ平台适配器
            version: 1.0.0
            author: Selrena Dev Team
            description: 适配NapCat框架的QQ平台消息接入插件
            main: dist/index.js
            permissions:
              - message.send
              - message.receive
              - network.access
            dependencies:
              "@cradle-selrena/protocol": "^1.0.0"
            minAppVersion: 1.0.0
            ```

    2. **沙箱隔离机制**：

        - 每个插件运行在独立的Node.js VM沙箱中，禁止访问原生`require`、`process`、`fs`、`net`等全局变量，仅能访问内核暴露的`kernelProxy`代理对象。

        - 插件的依赖独立安装在插件自身目录下，用pnpm workspace隔离，不会污染核心依赖，不会出现版本冲突。

        - 插件的内存、CPU使用有限额控制，超过限额自动触发告警、限制、甚至卸载，防止插件占用过多资源导致应用卡顿。

    3. **权限管控体系**：

        - 采用白名单权限体系，插件仅能使用清单中声明且用户授权的权限，无授权的API调用直接拦截，记录审计日志。

        - 权限分类全量定义：

|            权限分类|            核心作用|            授权级别|
|---|---|---|
|            `chat.*`|            对话能力相关权限，发送/接收消息、访问记忆|            用户手动授权|
|            `memory.*`|            记忆管理权限，读取/写入记忆|            用户手动授权|
|            `config.*`|            配置管理权限，读取/修改自身配置|            默认授权|
|            `plugin.*`|            插件管理权限，查看其他插件列表|            高级授权|
|            `native.*`|            原生算力调用权限，调用C++加速层|            高级授权|
|            `network.access`|            网络访问权限，发起HTTP请求|            用户手动授权|
|            `file.read`|            文件读取权限，读取指定目录文件|            用户手动授权|
|            `file.write`|            文件写入权限，写入指定目录文件|            高级授权|
        - 内核能力代理仅暴露与权限匹配的API，无权限的API在沙箱中完全不可见，从根源防止越权调用。

    4. **插件完整生命周期管理**：

|        生命周期阶段|        触发时机|        核心钩子|        内核操作|
|---|---|---|---|
|        加载|        应用启动/插件安装时|        `preLoad()`|        校验插件清单、初始化沙箱、注入内核代理|
|        初始化|        加载完成后|        `onInit()`|        校验插件初始化结果、注册插件事件监听|
|        启动|        初始化完成后|        `onStart()`|        启用插件功能、注册平台适配器、启动插件服务|
|        运行|        正常运行中|        `onEvent()`|        转发内核事件给插件、处理插件API调用|
|        停止|        应用停机/插件禁用时|        `onStop()`|        停止插件服务、释放资源、断开连接|
|        卸载|        插件删除时|        `onUninstall()`|        清理插件数据、移除沙箱、删除插件文件|
        - 所有生命周期钩子都是异步的，有超时控制，超时自动终止，防止插件阻塞内核启动/停机。

    5. **热插拔与热重载**：

        - 支持插件的热加载、热卸载、热重载，无需重启整个应用，修改插件代码后自动重载，提升开发效率。

        - 监听插件目录文件变化，自动检测插件更新，提示用户重载/更新插件。

    6. **崩溃隔离与异常处理**：

        - 插件的所有代码执行都有独立的异常捕获，插件抛出的异常不会影响内核和其他插件的正常运行。

        - 插件崩溃/异常次数超过阈值，自动停止插件，记录崩溃日志，通知用户，不会导致整个应用崩溃。

        - 插件异常不会污染内核上下文，沙箱完全隔离，异常后可直接重启插件，无需重启应用。

    7. **插件分类与管理**：

        - 插件分类全量定义，不同分类的插件有不同的权限范围和沙箱规则：

|            插件分类|            核心职责|            权限上限|            示例|
|---|---|---|---|
|            平台适配器插件|            多平台消息接入、消息标准化转换|            网络、消息收发|            QQ适配器、直播弹幕适配器、Discord适配器|
|            功能扩展插件|            核心功能扩展、工具能力增强|            记忆、配置、文件读取|            天气查询、备忘录、音乐控制、系统指令|
|            技能插件|            特定领域技能实现、专业能力扩展|            网络、原生算力、记忆|            代码编写、翻译、绘画、数据分析|
|            渲染扩展插件|            渲染层扩展、Live2D动作扩展、界面主题|            渲染API、文件读取|            Live2D形象包、界面主题、动作扩展包|
        - 插件启用/禁用、排序、优先级管理，高优先级的插件优先处理消息。

        - 插件市场兼容设计，预留在线安装、更新、卸载、评分的接口，支持后续扩展插件市场。

- **核心接口**：


- **边界铁律**：插件绝对不能直接访问内核内部实例、核心模块，仅能通过内核能力代理调用协议中定义的开放API；插件的所有操作都必须经过权限校验、参数校验、审计日志记录；插件绝对不能修改内核核心逻辑、全局配置。

##### 3.2.6 生命时钟管理器

**核心职责**：驱动月见的主动思维，实现「不需要用户说话也会自己思考、有自己的内心活动」的活着的特性，是数字生命的核心心跳。

- **核心功能**：

    1. 按全局配置定义的间隔，定期发送心跳给Python AI层，触发主动思维生成。

    2. 运行状态下，默认心跳间隔10s；休眠状态下，自动降频到60s，减少资源占用。

    3. 心跳异常自动重试，超过重试次数自动触发告警，不会导致内核崩溃。

    4. 主动思维结果自动同步给渲染层，驱动Live2D表情、动作变化，让用户直观感受到月见的内心活动。

    5. 支持特殊场景的心跳频率调整，比如用户长时间未交互时，自动降频，用户活跃时自动升频。

- **边界规则**：生命时钟仅由内核驱动，Python AI层仅响应心跳事件，不能自行修改心跳间隔、触发主动思维。

##### 3.2.7 全局事件总线

**核心职责**：实现内核内部模块间的解耦通信，模块间不直接调用，仅通过发布/订阅事件通信，消除硬编码依赖。

- **核心功能**：

    1. 异步非阻塞事件发布/订阅，单个事件处理器异常不会影响其他处理器执行。

    2. 所有事件必须继承全局协议中定义的事件基类，必须带trace_id，全链路透传。

    3. 事件订阅与模块生命周期绑定，模块停止时自动取消订阅，防止内存泄漏。

    4. 事件历史记录、审计日志，支持事件回溯、问题排查。

- **边界规则**：模块间不能直接调用对方的方法，必须通过事件总线通信；不允许发布未在全局协议中定义的私有事件。

##### 3.2.8 可观测性管理器

**核心职责**：全项目的日志、全链路追踪、Metrics指标、告警管理，保证所有流程可观测、问题可定位。

- **核心功能**：

    1. **结构化日志**：统一日志格式、级别、输出方式，所有日志都带模块、trace_id、时间戳，支持控制台输出、本地文件轮转存储、日志分级。

    2. **全链路追踪**：遵循OpenTelemetry规范，trace_id从用户输入到AI回复全流程透传，跨进程、跨模块、跨语言不中断，支持完整调用链查看。

    3. **Metrics指标**：采集内核运行状态、AI层响应时间、插件运行状态、内存/CPU使用率、模型推理耗时等核心指标，支持本地看板展示。

    4. **告警管理**：异常状态自动告警，比如AI层断开连接、插件崩溃、内存占用过高、模型加载失败，通过渲染层弹窗、日志、系统通知告知用户。

- **边界规则**：所有模块必须使用可观测性管理器输出日志、追踪、指标，不能自行打印控制台日志、写入本地文件。

##### 3.2.9 原生算力代理

**核心职责**：内核与C++加速层的唯一交互入口，封装C++原生算力能力，向上暴露标准化API，屏蔽底层硬件、平台差异。

- **边界规则**：内核其他模块、插件、Python AI层，绝对不能直接调用C++动态库，必须通过原生算力代理调用；C++层仅做纯算力运算，绝对不碰业务逻辑。

### 3.3 灵魂核心层（Python AI层，与历史代码100%对齐）

**核心定位**：月见的纯灵魂核心、OC意识本体，仅负责人格、记忆、情绪、思考、内容生成，完全封闭，仅与TS内核通过标准化IPC协议通信。

## 四、核心跨层机制详解

### 4.1 核心流程全链路时序（用户发消息的完整流程）

1. 用户在QQ发送消息 → QQ平台适配器插件接收消息，转换为内核标准化的`ChatMessage`事件，带唯一trace_id。

2. 插件通过内核能力代理，将标准化消息发送给内核对话API，内核执行权限校验、参数校验、审计日志记录。

3. 内核将消息存入短期记忆，更新用户熟悉度，通过IPC通信管理器，将标准化消息发送给Python AI层，透传trace_id。

4. Python AI层接收消息，执行完整思考流程：

    - 情绪系统基于用户输入更新情绪状态。

    - 从短期记忆中获取当前场景的对话上下文。

    - 检索相关的长期记忆、通用知识库。

    - 构建人设prompt，拼接所有上下文、记忆、知识。

    - 调用LLM引擎生成回复。

    - 执行人设边界红线校验，拦截违规内容。

    - 将用户输入和生成的回复沉淀到当前场景的短期记忆。

    - 发布`memory_sync`事件，将重要记忆同步给内核。

    - 发布`state_sync`事件，将当前情绪、状态同步给内核。

5. Python AI层将生成的回复通过IPC返回给内核，透传trace_id。

6. 内核接收回复，存入长期记忆库，持久化到数据库，同步状态给渲染层。

7. 渲染层接收状态同步，驱动Live2D表情、动作变化，展示对话气泡。

8. 内核将回复转发给对应的平台适配器插件，插件转换为对应平台的消息格式，发送给用户。

9. 全流程结束，所有日志、事件、操作都绑定trace_id，可完整回溯。

### 4.2 全链路可观测性体系

1. **结构化日志**：用winston（TS）和structlog（Python）实现，所有日志都带模块、trace_id、时间戳、级别，输出JSON格式，支持本地文件轮转存储、分级过滤。

2. **全链路追踪**：遵循OpenTelemetry规范，trace_id从用户输入生成，跨模块、跨进程、跨语言全链路透传，每个步骤都有span记录，支持完整调用链查看。

3. **Metrics指标**：采集核心指标，包括：

    - 应用指标：启动时间、内存/CPU使用率、运行时长。

    - AI指标：响应时间、推理耗时、主动思维生成次数、记忆召回数量。

    - 插件指标：插件数量、运行状态、崩溃次数、API调用次数。

    - 内核指标：IPC消息吞吐量、数据库操作耗时、事件总线处理耗时。

4. **告警管理**：定义告警规则，异常状态自动触发告警，通过渲染层弹窗、系统通知、日志告知用户，支持告警级别、静默规则配置。

### 4.3 错误处理与降级机制

1. **分层异常体系**：每个层有独立的异常类型，异常信息不会跨层泄露内部实现细节，所有异常都带错误码、trace_id、错误信息。

2. **崩溃隔离**：

    - 插件崩溃：自动停止插件，记录崩溃日志，通知用户，不会影响内核和其他插件。

    - Python AI层崩溃：内核自动重启AI子进程，恢复记忆和状态，用户无感知，不会导致应用崩溃。

    - 渲染层崩溃：内核自动重启渲染窗口，不会影响核心逻辑运行。

    - C++加速层异常：自动降级到CPU推理，或提示用户模型加载失败，不会导致应用崩溃。

3. **熔断降级**：

    - AI层响应超时：自动降级到预设的回复，不会卡死对话流程。

    - 插件调用超时：自动终止调用，返回超时错误，不会阻塞内核。

    - 模型推理失败：自动切换到备用模型，或降级到纯对话模式，保证基础功能可用。

4. **重试机制**：IPC通信异常、数据库操作失败、网络请求失败，自动重试，超过重试次数再报错，保证可靠性。

5. **死锁防护**：所有异步操作都有超时机制，所有资源都有释放保证，防止死锁、内存泄漏。

### 4.4 安全边界体系

1. **插件沙箱隔离**：VM沙箱隔离插件代码，禁止访问原生全局变量，仅能访问内核能力代理，所有API调用都经过权限校验。

2. **权限白名单机制**：所有权限都是白名单，插件仅能使用声明且授权的权限，无授权的操作直接拦截，记录审计日志。

3. **输入输出全校验**：所有外部输入、用户消息、插件调用参数，都经过格式校验、类型校验、安全过滤，防止注入攻击、越权操作。

4. **人设红线校验**：所有AI生成的内容，都经过人设边界红线校验，突破红线的内容直接拦截，不会发送给用户。

5. **数据加密**：敏感数据、用户记忆、配置，都加密存储到数据库，防止数据泄露。

6. **审计日志**：所有敏感操作、权限调用、数据修改，都记录审计日志，可完整回溯、追责。
