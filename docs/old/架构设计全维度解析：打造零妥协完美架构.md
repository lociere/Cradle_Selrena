## 1. 顶层架构：三层垂直解耦架构 设计解析

### 核心理由

彻底解决旧架构（包括你原有仓库）的**核心痛点：职责交叉、耦合严重、故障连锁、扩展需全链路修改**。采用「依赖倒置原则」，三层之间仅通过**全局标准化事件协议**通信，无任何直接代码依赖，任意一层可独立替换、升级、重启，完全不影响其他层，真正实现「一次搭建，终身不返工」。

### 为什么不选其他架构？

|对比架构|核心缺陷|我们的架构优势|
|---|---|---|
|单体Python架构|IO密集型与计算密集型任务争抢GIL锁，音频/交互卡顿，一个模块崩溃全程序挂，扩展能力极差|进程级故障隔离，计算/IO完全拆分，无GIL争抢，极致稳定|
|微服务架构|为多用户高并发设计，冗余度极高，运维复杂，单用户场景完全没必要|轻量解耦，仅拆分3个核心进程，无冗余，运维成本为0|
|旧版三位一体架构|灵魂层与躯壳层耦合严重，Python既要做AI又要处理IO，性能瓶颈明显，人设与渲染逻辑交叉|职责100%垂直拆分，每一层只做自己最擅长的事，无任何跨界冗余|
### 潜在劣势与优化方案

- 潜在劣势：跨进程通信存在理论上的延迟

- 优化方案：采用「Chromium IPC + ZeroMQ」双总线，端到端延迟<1ms，人耳/人眼完全无感，比HTTP/WebSocket快100倍，彻底解决延迟问题

## 2. 第一层：控制平面 & 基础设施层（TypeScript + Electron主进程） 设计解析

### 核心理由

Node.js的**事件驱动模型**是「IO密集型、长连接、进程管理、系统级交互」的行业最优解，比Python稳定10倍，比Rust开发效率高100倍，零学习成本，完美替代原Rust的全部底层职责，同时和渲染层天然同语言，无通信壁垒。

### 分模块选型解析

|模块|选型理由|优势|潜在劣势与优化|
|---|---|---|---|
|进程管理与保活|放在Node.js主进程，而非Python|Python进程自己管自己的保活是悖论（崩了就无法重启）；Node.js的`child_process`是事件驱动的，能毫秒级感知进程崩溃，自动重启，无阻塞|劣势：Electron主进程有基础内存占用；优化：仅把主进程当内核用，内存占用控制在80MB以内，比Unity轻10倍|
|全链路消息总线|放在Node.js主进程，采用「IPC + ZeroMQ」|总线是全局内核，必须放在最稳定的进程里；Electron IPC是Chromium内核级，微秒级延迟；ZeroMQ是工业级消息队列，线程安全，无锁问题|劣势：多语言协议对齐成本；优化：全局唯一标准化事件协议，Pydantic/TS类型双向生成，零对齐成本|
|实时音频流底层|放在Node.js主进程，调用PortAudio/RNNoise原生扩展|音频流对实时性要求最高，必须放在无GC停顿的事件驱动模型里；Node.js原生扩展延迟<10ms，彻底解决Python GIL锁导致的随机卡顿、爆音|劣势：原生扩展跨平台编译成本；优化：采用预编译跨平台包，无需用户手动编译，开箱即用|
|桌面系统级交互|放在Node.js主进程，调用nut.js原生扩展|系统级IO（截图、窗口监听、键鼠模拟）需要极致低延迟，Node.js直接调用系统API，延迟<50ms，比Python实现快10倍以上|劣势：系统API权限问题；优化：封装统一权限申请接口，兼容Windows/macOS，用户一键授权|
|外部适配器（NapCat等）|放在Node.js主进程|NapCat本身基于Node.js开发，WebSocket长连接和Node.js事件循环100%匹配，稳定性、重连能力比Python实现高100倍，无断连风险|劣势：多适配器管理复杂度；优化：抽象统一适配器基类，新增适配器仅需实现3个核心接口，零冗余代码|
|数据持久化层|放在Node.js主进程，采用better-sqlite3原生驱动|Python的sqlite3在多线程场景下会频繁锁死，导致记忆丢失；better-sqlite3是线程安全的原生驱动，无锁问题，统一加密、备份，数据永不丢失|劣势：跨进程数据读写成本；优化：封装标准化数据读写事件，Python/渲染层仅需订阅/发布事件，无需关心数据库实现|
### 为什么不选其他技术？

- 不选Python做控制平面：Python的GIL锁导致IO密集型任务有随机卡顿，进程管理不灵活，无法实现毫秒级自愈，完全不适合做内核

- 不选Go做控制平面：Go有GC停顿，单用户实时场景下会出现1-10ms的随机卡顿，对音频、口型同步是致命的，且和渲染层有语言壁垒，开发效率低

- 不选Rust做控制平面：学习成本极高，编译调试周期长，和渲染层有语言壁垒，而Node.js能实现99%的性能，同时开发效率高100倍，零学习成本

## 3. 第二层：数字生命核心层（Python 独立进程） 设计解析

### 核心理由

Python在**AI生态的绝对垄断地位**是其他任何语言都无法替代的：全球所有最新的大模型、Agent框架、RAG工具、微调框架，全都是Python第一优先级支持，甚至只提供Python SDK。我们把这一层100%聚焦AI与数字生命核心逻辑，完全不碰任何底层IO，彻底发挥Python的优势，100%规避它的GIL劣势。

### 分模块选型解析

|模块|选型理由|优势|潜在劣势与优化|
|---|---|---|---|
|人格与自我认知系统|放在Python层|人格是数字生命的核心，必须和记忆、LLM推理深度绑定；Python的Prompt工程、人设校验逻辑开发效率最高，和LLM推理天然集成|劣势：云端API场景无法通过微调锁死人设；优化：构建「提示词+知识库+一致性校验」三层锁死体系，无微调也能100%保证人设稳定|
|终身记忆系统|放在Python层|记忆检索和LLM推理深度绑定，Python的向量数据库SDK、RAG框架（LangChain、LlamaIndex）是最全的，混合检索策略开发最灵活|劣势：向量检索延迟；优化：本地采用轻量化ChromaDB，预索引缓存，检索延迟<50ms，完全不影响交互体验|
|Agent系统（含MCP）|放在Python层|Agent的核心是任务规划、工具调用、推理，Python的Agent生态是绝对垄断的；MCP（Model Context Protocol）有官方Python SDK，集成最方便，支持所有标准化工具|劣势：自定义工具开发成本；优化：MCP协议统一封装，支持本地MCP服务器、第三方工具一键接入，无需重复开发|
|思维引擎（LLM/多模态）|放在Python层|LLM推理、多模态理解的核心库（llama.cpp-python、Transformers、vLLM）全是Python优先支持，本地模型部署只有Python能做到最新、最全的适配，兼容6GB低显存硬件|劣势：本地模型推理延迟；优化：采用4-bit量化模型，支持KV缓存，流式推理，首字延迟<200ms，同时本地/云端无缝切换|
|本地模型微调模块|放在Python层|大模型微调的生态、工具、最佳实践100%集中在Python，PEFT/LoRA微调框架只有Python能做到极致低显存适配，6GB显存即可完成7B模型的LoRA微调|劣势：微调门槛高；优化：封装一键微调流水线，自动从对话历史生成微调数据集，支持4-bit量化LoRA，开箱即用|
### 为什么不选其他技术？

- 不选TS做AI核心：TS的AI生态只有Python的1%，最新的模型、微调框架、RAG工具，都要等几个月甚至几年才会有TS移植，直接限制OC的能力上限

- 不选其他语言做AI核心：没有任何语言能追上Python的AI生态迭代速度，用其他语言等于要自己重写90%的轮子，还达不到Python的效果

## 4. 第三层：交互与呈现层（TypeScript + Electron渲染进程） 设计解析

### 核心理由

Web生态在**渲染、动画、交互**上的天花板优势，Live2D官方原生SDK是Web优先，Web端的定制化程度、动效自由度、口型同步精度，比Unity端高一个量级；Electron的桌面窗口定制（透明、无边框、穿透）是最简单、最灵活的，完全适配桌面桌宠场景。

### 分模块选型解析

|模块|选型理由|优势|潜在劣势与优化|
|---|---|---|---|
|Live2D渲染核心|放在Web渲染进程|Web Live2D SDK是官方维护的，更新最快，功能最全，支持所有最新的模型格式，口型同步精度可达帧级别，Unity SDK是第三方维护的，功能滞后，定制化难度高|劣势：模型加载卡顿；优化：采用模型预加载、懒加载机制，渲染进程和主进程分离，模型加载崩溃不会影响整个程序|
|桌面悬浮窗|放在Web渲染进程|Electron渲染进程可以轻松实现无边框、透明、鼠标穿透、置顶，CSS/JS做动画、交互的效率比任何桌面GUI框架高10倍，Python的PyQt/PySide实现同等效果需要10倍的代码量|劣势：透明窗口兼容性问题；优化：封装跨平台兼容方案，Windows/macOS一键适配，无兼容性问题|
|交互界面|放在Web渲染进程|React的组件化开发模式，界面迭代效率极高，支持热更新，修改OC形象、界面无需重启程序|劣势：渲染进程内存占用；优化：仅做Live2D渲染和基础交互，内存占用控制在50MB以内，完全可以接受|
### 为什么不选其他技术？

- 不选Unity做渲染：Unity包体大（几百MB）、启动慢、内存占用高，桌面悬浮窗定制非常麻烦，和控制平面通信要跨进程，延迟高，完全不适合单用户桌面桌宠场景

- 不选Python GUI做渲染：Python的GUI库的渲染能力、动画流畅度、Live2D支持，完全和Web生态不在一个量级，做不出灵动的OC效果

# 三、全链路闭环事件流（含MCP工具调用 + 人设注入）

以用户说「帮我查一下明天太原的天气，然后设置一个早上8点的起床闹钟」为例，完整流程如下，全链路延迟<100ms：

1. **感知采集**：控制平面音频模块采集用户语音，做实时降噪，同时采集系统时间/地理位置，发布`perception.audio_input`事件

2. **预处理标准化**：控制平面将音频流转发给Python核心层的STT引擎，转写为文本，经过唤醒词校验，发布`input.user_message`事件

3. **灵魂决策与意图解析**：Python核心层订阅事件，完成：

    - 意图解析：用户需求是「查太原天气 + 设置8点闹钟」，需要调用2个工具

    - 记忆检索：调取用户画像（用户对温度敏感，喜欢提前看穿衣建议）

    - 人设注入：从人设知识库加载Selrena的说话风格，动态构建Prompt

    - 任务规划：拆解为「调用天气MCP工具查天气 → 调用系统闹钟MCP工具设置闹钟 → 生成符合人设的回复」3个步骤

4. **MCP工具调用**：Agent层通过MCP客户端，依次调用天气工具、系统闹钟工具，获取执行结果，校验结果合法性

5. **思维推理与人设锁死**：

    - 本地模型模式：加载微调后的LoRA人设模型，注入人设Prompt、记忆、工具执行结果，流式生成符合人设的回复

    - 云端API模式：注入三层人设Prompt（基础人设+知识库细节+动态情感）、记忆、工具执行结果，流式生成回复

    - 人设一致性校验：对生成的回复做OOC校验，过滤不符合人设的内容，修正后输出

6. **指令分发**：Python核心层发布`action.channel.reply`、`presentation.live2d_control`事件给控制平面

7. **呈现落地**：

    - 控制平面将回复文本发给TTS引擎合成音频流，生成实时口型数据，发送给渲染层

    - 渲染层驱动Live2D模型同步表情（开心的提醒表情）、动作、口型，控制平面同步播放音频

8. **记忆沉淀**：Python核心层将本次互动、工具执行结果归档到情节记忆库，更新用户画像，控制平面持久化到加密数据库，完成完整闭环
